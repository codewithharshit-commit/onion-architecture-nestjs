# Step 2: Create the RideEstimate Model

## What?

We will create a `RideEstimate` model.

This model represents the **result of estimating a ride**.

It contains:

- Pickup Location
- Drop Location
- Distance
- Duration
- Estimated Fare
- Ride Type

Example:

```text
Ride Estimate

Pickup  -> Delhi

Drop    -> Noida

Distance -> 18.5 km

Duration -> 35 min

Fare -> ₹285

Ride Type -> STANDARD
```

Notice that this is **not** a Ride.

No ride has been booked yet.

It is simply an estimate.

---

## Why do we need it?

Imagine our API returns:

```json
{
    "distance": 18.5,
    "duration": 35,
    "fare": 285
}
```

This works.

But what if tomorrow we also need

- Ride Type
- Currency
- Pickup
- Drop
- Surge Multiplier
- Toll Charges

We would keep changing our response everywhere.

Instead, we create one business object.

```text
RideEstimate
```

Everything related to fare estimation belongs inside it.

---

## Why isn't this an Entity?

An Entity has an identity.

Example:

```text
Ride

Ride Id

Driver Id

Rider Id
```

Every ride is unique.

Even if two rides have exactly the same pickup and drop,

they are still different rides because they have different IDs.

RideEstimate is different.

Suppose two users request

```text
Delhi

↓

Noida

↓

STANDARD
```

Both estimates will be identical.

There is no identity.

There is no database record.

Nothing is persisted.

Therefore,

**RideEstimate is a Domain Model, not an Entity.**

---

## Structure

```text
RideEstimate

Pickup Location

Drop Location

Distance

Duration

Ride Type

Estimated Fare
```

Notice that it is composed entirely of other domain objects.

```text
RideEstimate

│

├── Location

├── Location

├── Money

└── RideType
```

This is one of the strengths of Domain-Driven Design.

Complex business objects are built using smaller business objects.

---

## Why use Money instead of number?

Instead of

```typescript
fare = 250;
```

we store

```text
Money

Amount

Currency
```

Now RideEstimate doesn't care how money works.

Money already knows how to validate itself.

---

## Why use Location instead of latitude & longitude?

Instead of

```typescript
pickupLatitude;
pickupLongitude;

dropLatitude;
dropLongitude;
```

we simply use

```typescript
pickup: Location;

drop: Location;
```

The object becomes much cleaner.

---

## Responsibilities of RideEstimate

RideEstimate should only describe an estimated ride.

It should **not**

❌ Calculate pricing

❌ Call Google Maps

❌ Save to database

❌ Call APIs

Those responsibilities belong elsewhere.

---

## Domain Principle

A Domain Model represents a business concept.

It should contain only the information required to describe that concept.

Business Logic

↓

PricingService

↓

creates

↓

RideEstimate

↓

Application Layer returns it

---

## Summary

After this step our Domain looks like

```text
Domain

│

├── Value Objects
│     ├── Location
│     └── Money
│
├── Enum
│     └── RideType
│
└── Model
      └── RideEstimate
```

We now have all the objects needed to describe a fare estimate.

The next step is to build the **PricingService**, which contains the actual business rules for calculating the fare.

# Step 3: Create the PricingService

## What?

The `PricingService` is a **Domain Service** responsible for calculating the estimated fare of a ride.

Given:

- Distance
- Duration
- Ride Type

it returns:

- Estimated Fare

Example

```text
Distance = 18 km

Duration = 35 min

Ride Type = STANDARD

↓

Fare = ₹305
```

---

## Why do we need a Domain Service?

Ask yourself this question.

> **Who should know how to calculate the fare?**

Certainly not

- Controller ❌
- Database ❌
- Google Maps ❌
- NestJS ❌

The pricing rules belong to the business itself.

Therefore they belong inside the **Domain Layer**.

---

## Why not put this logic inside RideEstimate?

Because `RideEstimate` only represents the result.

Think of it like this.

```text
Invoice

↓

Represents an invoice
```

Does an Invoice calculate taxes?

Usually no.

A separate service creates it.

Similarly,

```text
RideEstimate

↓

Represents an estimated ride
```

The object describing the estimate shouldn't know how to create itself.

That responsibility belongs to `PricingService`.

---

## Responsibilities of PricingService

It should know:

✔ Base Fare

✔ Cost per Kilometer

✔ Cost per Minute

✔ Minimum Fare

✔ Ride Type Pricing

✔ Surge Pricing (future)

✔ Airport Charges (future)

✔ Toll Charges (future)

✔ Promo Discounts (future)

---

It should NOT know

❌ Google Maps

❌ Database

❌ HTTP APIs

❌ Redis

❌ NestJS

Those belong to other layers.

---

## First Pricing Formula

We'll keep it simple.

```text
Fare

=

Base Fare

+

(Distance × Rate per KM)

+

(Duration × Rate per Minute)
```

Example

```text
Base Fare = ₹50

Distance = 18 km

Rate/km = ₹10

Duration = 35 min

Rate/min = ₹2

Fare

=

50

+

180

+

70

=

₹300
```

This formula is purely business logic.

That's exactly what belongs inside the Domain.

---

## Why not hardcode numbers in the Controller?

Bad

```text
Controller

↓

fare = 50 + distance * 10
```

Tomorrow

Business says

> Increase base fare to ₹60.

Now every place using that logic must change.

Instead,

```text
Controller

↓

PricingService

↓

Fare
```

Only one class changes.

---

## Future Growth

Today

```text
Fare

=

Base

+

Distance

+

Duration
```

Tomorrow Uber might say

```text
Fare

=

Base

+

Distance

+

Duration

+

Surge

+

Airport Fee

+

Toll

-

Promo
```

Nothing changes outside the PricingService.

The rest of the application doesn't even know the pricing rules changed.

---

## Domain Principle

The PricingService contains **business policies**.

Business policies change frequently.

Infrastructure changes occasionally.

Frameworks change even more often.

By isolating business policies inside the Domain, the core of the application remains independent of external technologies.

---

## Summary

Current Domain

```text
Domain

│

├── Value Objects
│     ├── Location
│     └── Money
│
├── Enum
│     └── RideType
│
├── Model
│     └── RideEstimate
│
└── Domain Service
      └── PricingService
```

The PricingService is the first class that contains actual business rules.

Everything we've built so far exists to support this service.

# Step 4: Create the RouteProvider Interface (Port)

## What?

We will create a `RouteProvider` interface.

Its job is simple.

Given:

- Pickup Location
- Drop Location

Return:

- Distance
- Duration

Example

```text
Pickup

↓

Delhi

Drop

↓

Noida

↓

RouteProvider

↓

Distance = 18.5 km

Duration = 35 min
```

Notice that it does **not** say **how** the distance is calculated.

---

## Why do we need it?

Our PricingService needs:

```text
Distance

Duration
```

Someone has to calculate them.

Possible providers are:

- Google Maps
- Mapbox
- HERE Maps
- OpenRouteService
- OSRM

Should our Domain know which one is used?

No.

The business doesn't care.

It only cares about receiving:

```text
Distance

Duration
```

---

## Why an interface?

Imagine this.

Without an interface:

```text
PricingService

↓

Google Maps API
```

Now Google Maps becomes part of our Domain.

Tomorrow, if the company decides to switch to Mapbox because it's cheaper:

```text
Google Maps

↓

Mapbox
```

We would have to modify our Domain.

That violates Onion Architecture.

Instead, the Domain depends on an abstraction.

```text
PricingService

↓

RouteProvider

↓

Google Maps
```

Tomorrow:

```text
PricingService

↓

RouteProvider

↓

Mapbox
```

The Domain doesn't change at all.

Only the Infrastructure implementation changes.

This follows the **Dependency Inversion Principle**.

---

## What should the interface return?

Instead of returning two primitive values:

```typescript
distance;

duration;
```

we'll create another small Domain Model.

```text
Route

Distance

Duration
```

Why?

Because distance and duration always belong together.

Whenever someone asks for a route,

they need both.

This keeps the code more expressive.

---

## Responsibilities

RouteProvider should only define a contract.

It should **not**

❌ Call Google Maps

❌ Call HTTP APIs

❌ Parse JSON

❌ Handle Authentication

Those responsibilities belong to Infrastructure.

---

## Domain Principle

The Domain defines **what it needs**.

Infrastructure decides **how to provide it**.

```text
Domain

↓

"I need distance."

↓

Infrastructure

↓

"I'll get it from Google Maps."
```

This is one of the core ideas behind Onion Architecture.

---

## Summary

Our Domain now contains:

```text
Domain

│

├── Value Objects
│     ├── Location
│     └── Money
│
├── Enum
│     └── RideType
│
├── Models
│     ├── RideEstimate
│     └── Route
│
├── Domain Service
│     └── PricingService
│
└── Interface
      └── RouteProvider
```

The Domain is now complete enough to describe **what** it needs to estimate a fare, while remaining completely independent of **how** that information is obtained.
